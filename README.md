# REST-сервис для работы с постами и комментариями

## Вопросы
### 1.	Какие узкие места появятся при 100 000 постов и 1 000 000 комментариев?
В среднем 10 комментариев на пост. В первую очередь увеличится нагрузка на БД. Для запросов на чтение необходимы соответствующие индексы, иначе будет проводится времязатратное сканирование таблиц.

Также могут возникнуть проблемы с задачей обновления статуса комментария через 30 секунд  после его создания. Независимо от выбранного решения большое количество комментариев может привести к одновременной работе нескольких отложенных задач. Возникает необходимость в блокировках или в увеличении интервала запуска отложенных задач (что увеличит возможную задержку обновления статуса комментариев, но даст больше времени на выполнение задачи).

### 2.	Как бы вы изменили архитектуру для масштабирования?
Можно добавить кеширование списка постов с обновлением. Дополнительно кешировать на короткий промежуток времени(несколько минут) отдельные посты с комментариями при наличии недавних изменений.

Перенести задачу по смене статуса комментария через 30 секунд после публикации с rest-сервиса в базу данных, что уже реализовано в данном решении. Это позволит убрать проблему выполнения одного задания на параллельно запущенных копиях rest-сервиса. Таким образом можно запускать множество облегченных rest-сервисов одновременно. Дополнительно к этому можно запустить балансировщик нагрузки и очередь для создания комментариев.

### 3.	Какие индексы вы считаете необходимыми и почему?
Необходимы индексы, покрывающие запросы к БД.

**К таблице постов:**

(is_deleted, created_at) - для сортировки по дате (только неудалённые)

(is_deleted, com_counter) - для сортировки по количеству комментариев

**К таблице комментариев:**

(post_id, status) - для  получения всех одобренных комментариев поста

(parent_comment_id) - для поиска родительского комментария

(id, status) - для поиска одобренных комментариев

### 4.	Возможны ли проблемы при одновременном создании комментариев? Если да - какие?
При одновременном создании множества комментариев к одному посту может возникнуть очередь для поиска поста и проверки уровня вложенности.

Текущие ограничения не позволят создать комментарий к несуществующему посту или создать ответ к несуществующему или несоответствующему посту комментарию.

Но в случае высокой нагрузки при создании комментариев есть возможность мягкого удаления поста в процессе создания комментариев. Тогда комментарии будут созданы для уже удаленного поста.

## Функционал
1. Создание поста: POST http://адрес:порт/api/posts -H "Content-Type: application/json" -d '{"title":"Заголовок","content":"Текст поста"}'

2. Получение списка постов: http://адрес:порт/api/posts

    1. Список постов по дате: http://адрес:порт/api/posts?sortBy=createdAt

    2. Список постов по кол-ву комментариев: "http://адрес:порт/api/posts?sortBy=comCounter"

3. Получение конкретного поста с комментариями: http://адрес:порт/api/posts/id поста

4. "Мягкое" удаление поста: DELETE http://адрес:порт/api/posts/id поста

5. Создать комментарий к посту: POST http://адрес:порт/api/comments -H "Content-Type: application/json" \-d '{"text":"Текст комментария","postId":id поста}'

6. Создать ответ к комментарию: POST http://адрес:порт/api/comments -H "Content-Type: application/json" -d '{"text":"Ответ","postId":id поста,"parentCommentId":id родительского комментария}'




## Задача по обновлению статуса комментариев в pg_cron + увеличение счетчика комментариев поста
```sql
SELECT cron.schedule_in_database(
    'approve-comments-and-counter',
    '5 seconds',
    $$
    WITH pending AS (
        SELECT id, post_id
        FROM comments
        WHERE status = 'PENDING'
          AND created_at <= NOW() - INTERVAL '30 seconds'
        FOR UPDATE SKIP LOCKED                              --блокировщик на случай запуска нового задания до завершения старого
    ),
    updated_comments AS (
        UPDATE comments c
        SET status = 'APPROVED'
        FROM pending p
        WHERE c.id = p.id
        RETURNING c.post_id
    ),
    post_counts AS (
        SELECT post_id, COUNT(*) AS cnt
        FROM updated_comments
        GROUP BY post_id
    )
    UPDATE posts p
    SET com_counter = com_counter + pc.cnt
    FROM post_counts pc
    WHERE p.id = pc.post_id;
    $$,
    'имя бд'   
);
```
Стоит добавить задачу для очиски истории pg_cron
```sql
SELECT cron.schedule(
    'cleanup-job-history',
    '0 0 * * 0',
    $$DELETE FROM cron.job_run_details WHERE end_time < NOW() - INTERVAL '7 days'$$
);
```
Pg_cron требует ручной установки и настройки, поэтому я не включил его в скрипты flyway.

Устанавливать pg_cron должен суперпользователь. Лично у меня возникла проблема с failed connection при выполнении отложенных задач. Решить проблему удалось командой ```UPDATE cron.job SET nodename = '';```
